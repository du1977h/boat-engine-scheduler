# インシデント整理メモ: 不正侵入・クリプトジャッキング事案から学ぶこと

## 1. この文書の目的

この文書は、今回の VPS 上で確認された高 CPU 使用率事案について、

- 何が起きていたのか
- そこから何を学ぶべきか
- 今後どう防ぐべきか
- 直近で何を対応すべきか

を、運用担当者向けに整理したものです。

今回の事案は、単なる `nginx` や HTTPS の負荷問題ではなく、サーバー侵害後に暗号通貨マイナーや永続化処理が仕込まれていた可能性が高い、重大なセキュリティインシデントとして扱うべき内容です。

---

## 2. 今回確認された事象

今回の調査で確認できた主な事実は以下の通りです。

### 2.1 高負荷の正体

`top` の結果では、CPU を大きく消費していた主因は `nginx` ではなく、`https` という名前の別プロセスでした。

確認された実行内容の例:

```text
./https -a rx/0 -o pool.supportxmr.com:3333 ...
```

これは HTTPS サーバー処理ではなく、暗号通貨マイナーの典型的な起動パターンです。

### 2.2 クリプトジャッキングの兆候

以下の要素から、今回の事案は `cryptojacking`、つまり「サーバー侵害後に仮想通貨マイナーを設置して計算資源を不正利用する攻撃」である可能性が高いです。

- `pool.supportxmr.com:3333` への接続
- `rx/0` の指定
- `https` という正規プロセスに見せかける名前
- `root` 権限での起動

### 2.3 永続化の痕跡

`crontab` には、不正な永続化処理とみられる設定がありました。

例:

- 30 分ごとに `pakchoi` ユーザー作成を試行
- `NOPASSWD:ALL` を sudoers に追加しようとする処理
- `@reboot` で `/var/tmp/systemd-logind` を起動
- 外部サーバーから `sync.x86_64` を取得して実行する処理
- `/home/du/.local/bin/.X11-linux` を定期的に再起動する処理

これらは、攻撃者が「再起動しても復活する」ように仕掛ける典型的な永続化手口です。

### 2.4 nginx / HTTPS 自体も負荷の見え方に影響

アプリ停止後も `nginx` は 443/TCP で待ち受けていたため、停止した upstream に対して `502 Bad Gateway` を返していました。

そのため、監視画面上では「HTTPS の負荷」に見えても、実際には以下が混ざっていた可能性があります。

- `nginx` による TLS 終端処理
- ブラウザや bot からのアクセス継続
- 停止中 upstream への接続失敗
- `https` という名前に偽装したマイナー

つまり、「HTTPS が重い」のではなく、「`https` という名前の不正プロセスが重い」ことが本質でした。

## 3. この事例から学ぶべきこと

### 3.1 プロセス名を信用しすぎない

`https` という表示だけを見ると、HTTPS 通信や `nginx` の処理に見えてしまいます。
しかし実際には、正体は別の実行ファイルでした。

学ぶべき点:

- `top` の `COMMAND` 名だけで判断しない
- `ps -fp <PID>`、`/proc/<PID>/cmdline`、`/proc/<PID>/exe` で正体を確認する
- 「名前がそれっぽい」だけのプロセスを正規のものと思い込まない

### 3.2 高 CPU は性能問題ではなく侵害の兆候でもある

CPU 高騰を見たとき、多くの場合はまずアプリや DB、`nginx` 設定を疑います。
しかし今回は、原因は性能問題ではなくセキュリティインシデントでした。

学ぶべき点:

- 高負荷時は性能と侵害の両面で切り分ける
- 不審な外向き通信先も確認する
- `cron` や `systemd` の異常も見る

### 3.3 アプリ停止だけでは止まらない負荷がある

今回のように、アプリ本体を止めても以下は残り続けます。

- `nginx` の待受
- bot からのアクセス
- 停止した upstream への再試行
- マイナーやバックドアなど別プロセス

学ぶべき点:

- 「アプリを止めたのに負荷が下がらない」ときは周辺プロセスを調べる
- Web サービスとサーバー全体を分けて考える

### 3.4 永続化を仕込まれると単純停止では解決しない

不正プロセスを kill しても、`cron` や `@reboot` が残っていれば復活します。

学ぶべき点:

- `kill` は応急処置にすぎない
- `crontab`、`systemd`、`/etc/sudoers.d`、`~/.ssh` などの永続化ポイントを確認する
- 完全性が疑われるサーバーは再構築前提で扱う

### 3.5 Web 経由の侵入可能性を常に考える

今回、`/tmp/sync.x86_64` が `www-data` 所有で作成されていた点は重要です。
これは、Web 経由で何らかのファイル取得や実行が行われた可能性を示唆します。

学ぶべき点:

- SSH だけでなく Web アプリも侵入口になりうる
- 同居アプリがある VPS は、どれか 1 つの脆弱性で全体が危険になる
- 公開サービスごとの隔離と権限制限が重要

## 4. 想定される侵入経路

現時点では断定できませんが、一般に今回のような事例で考えられる侵入経路は以下です。

### 4.1 SSH 認証情報の漏えい

- パスワードの使い回し
- 弱いパスワード
- 秘密鍵の漏えい
- root ログイン許可

### 4.2 公開 Web アプリの脆弱性

- 古い CMS、プラグイン、管理画面
- ファイルアップロード処理
- コマンド実行系の脆弱性
- 未更新の npm / PHP / Python 依存

### 4.3 不審なセットアップスクリプトの実行

- `curl ... | sh`
- 出所不明のバイナリ
- 検証なしの install スクリプト

### 4.4 同居サービスからの横展開

1 台の VPS に複数アプリが載っている場合、別アプリ側の脆弱性や設定不備から侵入されることがあります。

### 4.5 権限昇格

最初は一般ユーザー権限や `www-data` 権限だったとしても、その後 root 権限に昇格された可能性があります。

## 5. 今後の予防方法

### 5.1 SSH 防御を最優先で強化する

必須に近い設定:

- `PasswordAuthentication no`
- `PermitRootLogin no`
- 公開鍵認証のみ許可
- sudo 可能ユーザーを最小化
- SSH ログイン元を可能なら IP 制限

推奨:

- `fail2ban` 導入
- SSH ポートの変更は補助策として活用

### 5.2 公開ポートを最小化する

原則として外部公開は次に限定します。

- `80`
- `443`
- `22`

それ以外はファイアウォールで閉じます。
Node.js アプリや SQLite、補助サービスは `127.0.0.1` バインドを徹底します。

### 5.3 OS とミドルウェアを定期更新する

対象:

- Ubuntu パッケージ
- `nginx`
- Node.js
- npm 依存パッケージ
- 併設アプリ

推奨:

- `unattended-upgrades`
- 月次の手動アップデート確認
- 脆弱性アラート確認

### 5.4 サービスごとに権限を分離する

避けるべき状態:

- アプリ実行ユーザーに sudo 権限がある
- 複数アプリが同じユーザーで動く
- 書き込み可能ディレクトリが広すぎる

望ましい状態:

- アプリごとに専用ユーザー
- systemd サービスも専用ユーザーで実行
- 書き込み権限は必要最小限
- `/tmp` やホームディレクトリから実行されるものを警戒

### 5.5 出所不明のコードを root で実行しない

非常に重要です。

- `curl | sh` を安易に実行しない
- GitHub の不明なスクリプトを root で動かさない
- `wget` したバイナリをそのまま実行しない

導入手順は公式ドキュメントベースに絞るのが安全です。

### 5.6 永続化ポイントを定期監査する

少なくとも以下は定期確認対象にします。

- `crontab -l`
- `/etc/crontab`
- `/etc/cron.*`
- `/etc/systemd/system`
- `/etc/sudoers.d`
- `/var/tmp`
- `/tmp`
- `~/.ssh/authorized_keys`
- 不審ユーザーの存在

### 5.7 監視を導入する

最低限ほしい監視:

- CPU 使用率急上昇
- メモリ使用量急増
- 不審な外向き通信先
- 不審プロセス名
- root の `cron` 変更
- `sudoers` 変更

可能なら以下も有効です。

- `auditd`
- `AIDE`
- `rkhunter`
- `chkrootkit`

### 5.8 同居アプリを減らす、または隔離する

複数アプリを 1 台に載せるなら、最低限以下を徹底します。

- 実行ユーザー分離
- 設定ファイル分離
- systemd サービス分離
- nginx upstream 分離
- アプリごとの権限とデータ領域分離

より安全にするなら、用途ごとに VPS を分けるのが理想です。

### 5.9 バックアップ戦略を持つ

今回のような侵害では、最終的に再構築が最善になることが多いです。

そのため、日頃から最低限以下を別保管しておきます。

- SQLite DB
- `.env` など環境設定
- サンプル CSV ではない本番 CSV
- nginx 設定
- systemd ユニット
- アプリ本体

バックアップは「取る」だけでなく「復元テスト」を行うことが大切です。

## 6. 今後の対応方針

### 6.1 直ちに行うべき対応

1. サーバーを信頼しない前提に切り替える
2. ネットワーク的に隔離するか、公開停止する
3. 不正プロセスを停止する
4. 不正 `cron`、不審 `systemd`、不審ファイルを確認する
5. すべての秘密情報を漏えい前提でローテーションする

ローテーション対象の例:

- SSH 鍵
- パスワード
- GitHub / npm / 各種 API トークン
- メール認証情報
- クラウド認証情報

### 6.2 中期対応: 再構築を前提にする

今回のように root 権限を含む侵害の可能性がある場合、原則として「掃除して使い続ける」より「再構築」が安全です。

推奨手順:

1. 正規データのみ退避
2. VPS を再作成
3. OS 初期設定をやり直す
4. SSH / firewall / fail2ban を先に適用
5. その後にアプリを再配置
6. バックアップから必要最小限のデータを復元

### 6.3 退避時の注意

退避すべきもの:

- DB ファイル
- 設定ファイル
- アプリソースコード
- アップロード済み正規データ

退避に注意すべきもの:

- `/tmp`、`/var/tmp`
- 実行ファイル
- 不審な `.sh`
- 不審なバイナリ
- `cron` や `systemd` の不審設定

「動いていたものを丸ごと戻す」のは避けるべきです。

### 6.4 事後レビューを行う

可能であれば以下を残します。

- 発見日時
- 兆候
- 影響範囲
- 判明した IOC
- 実施した封じ込め
- 再発防止策

IOC の例:

- `pool.supportxmr.com`
- `pakchoi`
- `sync.x86_64`
- `systemd-logind` in `/var/tmp`
- `.X11-linux`
- `https` という偽装バイナリ

## 8. 今回の事案についての結論

今回のケースでは、単なるアプリや HTTPS の設定不備として扱うのは危険です。

確認できた内容からは、

- サーバー侵害
- クリプトジャッキング
- 永続化
- バックドア設置の試行

が疑われます。

したがって、最も重要な判断は次です。

> このサーバーは「一度侵害された可能性が高い資産」として扱い、継続利用ではなく再構築を第一選択とする。

---

## 9. 補足: 実務上の優先順位

優先順位は以下を推奨します。

1. 被害拡大の停止
2. 認証情報のローテーション
3. 正規データの退避
4. VPS 再構築
5. 再発防止設定
6. 事後レビュー

侵入経路の厳密な特定は重要ですが、それよりも「今後このサーバーを信頼しない」判断のほうが優先度は高いです。

---

## 10. ひとことでまとめると

今回学ぶべき最重要ポイントは次の 4 点です。

- 高 CPU は性能問題ではなく侵害の兆候でもある
- プロセス名や見た目に騙されず正体を確認する
- `kill` だけでは不十分で、永続化の除去か再構築が必要
- VPS は「最小公開」「最小権限」「監視」「再構築前提」で運用する
